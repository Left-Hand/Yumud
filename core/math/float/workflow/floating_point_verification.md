# 浮点数格式转换的形式化验证工作流

## 概述

本工作流适用于**位操作密集的转换函数**（浮点数格式转换、编解码、位打包等）。通过系统化验证发现隐蔽 bug。

核心思想：用不同的验证方法**交叉检验**，使 bug 无处遁形。

---

## 工作流步骤

### 1. 前向转换验证

**目标**：验证 `A → B` 转换的正确性

**方法**：
- **约束求解**：用 Z3 等 SAT 求解器检验约束的满足性
- **边界值测试**：覆盖零值、最小/最大、溢出/下溢、特殊值
- **特性验证**：符号保留、单调性、范围映射

**输出**：✓ 转换代码正确 / ✗ 发现 bug

---

### 2. 反向转换验证

**目标**：验证 `B → A` 转换的正确性

**方法**：
- **分类验证**：按格式规范分类（零值、规范化数、非规范化数、特殊值）
- **结构化测试**：每类至少 1-2 个测试用例
- **逐值对比**：原始实现 vs 修正实现的差异分析

**输出**：✓ 反向转换正确 / ✗ 诊断 bug 类型

---

### 3. 往返转换一致性

**目标**：验证 `A → B → A` 的精度保证

**方法**：
- **等值检验**：某些值应完全恢复（如整数、2 的幂）
- **精度检验**：其他值允许有界误差（如相对误差 < 1%）
- **范围测试**：覆盖有效范围的各个区间

**输出**：✓ 往返一致 / ✗ 确定精度损失来源

---

### 4. Bug 诊断

**触发条件**：往返转换失败时

**诊断法**：
1. 实现一个**参考实现**（简单但正确的版本）
2. 逐值对比原始与参考实现
3. 分析差异模式，找出根因

**常见 bug 类型**：
- 位操作错误（如多余的位设置、移位方向错误）
- 指数/尾数映射规则漏缺
- 特殊值处理不完整
- 边界条件判断错误

**输出**：根因分析 + 修复建议

---

### 5. 修复与合并

**修改**：
1. 修正源代码
2. 更新文档（添加验证参考）
3. 补充单元测试

**验证**：
- 编译期测试（`static_assert`）
- 重新运行形式化验证脚本

**输出**：修复完成，所有验证通过

## 关键决策点

| 问题 | 选择 | 原因 |
|------|------|------|
| 何时用 Z3？ | 约束满足性、范围验证 | 数学证明，覆盖无穷情况 |
| 何时用单元测试？ | 具体值、往返验证 | 快速反馈，易于调试 |
| 如何选边界值？ | 格式的零点、极值、转折点 | 最容易暴露 bug |
| 精度容限设多少？ | 格式精度的 1-10% | 根据应用场景调整 |

## 验证脚本结构

```
verify_FORMAT_complete.py
  ├─ 阶段 1：前向转换验证
  │   ├─ 约束求解 / 边界值测试
  │   └─ 输出：通过/失败
  ├─ 阶段 2：反向转换验证
  │   ├─ 分类测试 / 逐值对比
  │   └─ 输出：通过/失败
  └─ 阶段 3：往返一致性
      ├─ 等值检验 / 精度检验
      └─ 输出：通过/失败
```

## 应用场景

✓ 跨平台兼容性验证
✓ 精度要求严格的数学库

## 最佳实践

1. **尽早验证**：设计确认后立即编写验证脚本
2. **分层验证**：约束 → 边界 → 往返，逐层深入
3. **保留脚本**：用作回归测试和新修改的baseline
4. **文档化 bug**：记录发现的 bug 及修复，建立 bug 库
5. **自动化**：集成到 CI/CD，每次代码变更自动运行

---

## 工作流优势

| 传统单元测试 | 形式化验证工作流 |
|------------|--------------|
| 需要手写所有测试用例 | 自动生成/枚举测试用例 |
| 难以发现隐蔽 bug | 用数学约束保证正确性 |
| 仅验证已想到的场景 | 穷尽式验证（Z3） |
| 跨格式转换难以验证 | 系统化方法应用任何格式 |

---

## 常见问题

**Q: 验证脚本很慢？**
A: 减少 Z3 求解的约束复杂度，或用参考实现替代部分验证

**Q: 如何处理浮点精度问题？**
A: 用相对误差容限而非等值检验；明确文档化精度预期

**Q: 发现 bug 后如何修复？**
A: 参照工作流阶段 4：修复代码 → 补充单元测试 → 重新验证

---

## 总结

形式化验证工作流的核心：**多角度交叉验证**

- 约束求解 → 数学保证
- 边界值测试 → 覆盖极端情况
- 往返一致性 → 实际可用性

通过这三层验证，将 bug 消灭在代码评审阶段。
